# REQ-20260502-002 升级知识库检索链路、多轮上下文与默认新对话策略

## 背景

当前知识库虽然已经具备基础混合检索能力，但在真实问答里仍有几个明显短板：

1. 文档标题命中弱，标题里有、正文里没有的主题常常找不到。
2. 命中结果过碎，只拼接零散片段，缺少同章节上下文。
3. 追问理解依赖较弱，用户不重复主语时容易答偏。
4. 知识库页会自动恢复最近一条历史，旧话题容易在新进入页面时干扰当前问题。

## 目标

1. 在现有 `SQLite + sqlite-vec + FTS5` 架构内，把知识库检索升级为“查询规划 + 多路召回 + 章节证据扩展”。
2. 提升标题、路径、文件名相关问题的命中率，减少“文档明明存在却回答不知道”。
3. 让追问更稳定地继承上一轮语义，同时避免串到旧会话。
4. 保持知识库页历史可查看、可恢复，但页面默认进入新对话。

## 结果

- 已新增查询规划层：支持 `intent`、`standalone_query`、`expanded_query`、`keywords`、`title_hints`。
- 已新增文件级 `files_fts` 标题检索，并将标题命中的文件纳入二次 chunk 召回。
- chunk 检索文本与向量输入已统一带入文档标题、文件路径、标题层级和正文。
- 检索结果已从离散 `chunks` 扩展为带上下文的 `sections` 证据包，回答优先依据章节证据作答。
- 当只命中文档标题、正文证据仍偏弱时，系统会明确提示“已定位到相关文档，但正文证据有限”，不再直接空泛回复“不知道”。
- 知识库页首次进入时不再自动恢复最近一条历史；历史仍保留在抽屉中，用户主动选择后才续聊。
