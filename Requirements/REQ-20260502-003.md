# REQ-20260502-003 升级知识库单索引问答链路并增加成本护栏

## 背景

当前知识库已经具备“查询规划 + 混合检索 + 章节证据扩展”的主链路，但继续提高回答准确性时，仍然有几个问题没有解决：

1. 用户追问不够明确时，系统缺少稳定的澄清分支，容易直接检索后答偏。
2. 有些问题需要轻量改写或重排才能答稳，但如果每轮都多调一次模型，成本会快速上升。
3. 证据偏弱、证据冲突、完全无证据这三类情况没有被清楚地区分，前后端也没有统一元数据口径。
4. 现有链路缺少针对 helper 调用、弱证据和冲突识别的回归样例与指标。

## 目标

1. 保持单索引架构，不新增文档摘要索引、章节摘要索引和句级持久索引。
2. 在回答质量优先的前提下，把业务级 LLM 调用收敛到“辅助最多 1 次 + 主回答最多 1 次”。
3. 让知识库问答明确区分 `clarify_needed / grounded / weak_evidence / conflicting_evidence / no_evidence` 五种回答模式。
4. 为查询理解、辅助缓存、消息元数据、可观测日志和知识库回归测试补齐统一实现。

## 结果

- 查询规划新增 `clarity_score`、`ambiguity_flags`、`clarify_needed`、`clarify_question`、`rewrite_strategy`，并按阈值区分“直接检索 / 改写 / 追问”。
- 新增知识库 helper 短时缓存，固定覆盖 `rewrite` 与 `rerank`，缓存键包含会话、查询、当前文档、手动参考和历史摘要哈希，TTL 为 5 分钟。
- `/api/chat` 现在支持 `clarify_needed` 和 `no_evidence` 模板化直返，这两类场景不会再调用主回答模型。
- 单次请求被限制为最多一次辅助调用；同一轮不会同时触发 `rewrite + rerank`。
- 检索链路补充更大的 chunk/section 候选池、`evidence_sentences`、`top_score_gap`、`section_file_count`，并为条件重排提供输入。
- 新增知识库运行时模块，负责 prompt 压力判断、条件 rerank、弱证据原因、冲突摘要和回答模式判定。
- `messages` 表已支持 `meta` 字段，落库存储回答模式、清晰度、歧义标记、重排状态、检索统计和 helper 遥测。
- 知识库页前端已改为消费 `assistant_meta / done.meta`，显示真实回答模式，不再用“是否有 citations”粗略推断。
- 新增知识库回归样例、路由测试和检索测试，并输出 `Hit@K`、`MRR`、`clarify_precision`、`weak_evidence_precision`、`conflict_detection_rate`、`false_grounded_rate`、`helper_trigger_rate`、`avg_llm_calls_per_request`。
