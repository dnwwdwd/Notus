# REQ-20260502-005 升级创作风格仿写与改写范围识别

## 分类

- 类型：功能优化
- 状态：已完成

## 背景

原有创作页的风格仿写主要依赖运行时临时检索到的少量原文片段，范围识别也主要依赖 `@bN` 显式指定。这样会带来几个问题：

1. 风格参考更像“主题相似片段”，不是稳定的个人风格建模。
2. 用户自然语言里说“改前两段”“统一全文语气”“继续改刚才那段”时，系统不能稳定识别改单块、多块还是全文。
3. AI 预览只支持单条待应用修改，刷新后无法恢复多条预览。
4. 全文改写缺少预算上限和分批策略，容易在效果和成本之间失衡。

## 本次目标

1. 在不新增独立风格向量索引的前提下，显著提升风格仿写稳定性。
2. 让创作链路自动识别单块、多块、全文改写范围，并继续兼容 `@bN` 指定。
3. 让批量预览可持久化、可恢复、可整体应用或取消。
4. 给全文改写加上固定上限、分批执行和降级规则，优先保证准确性。

## 实施结果

### 1. 风格建模升级

- 新增 `style_fingerprints` 表，按文章保存风格指纹：
  - `file_id`
  - `file_hash`
  - 五维风格特征
  - `signature_phrases_json`
  - `status / retry_count / last_error / model_used`
- 新增 `style_profile` 表，保存全局最新风格画像。
- 仅当文章满足 `>= 3` 个可写文字块且正文长度 `>= 600` 字时才参与风格提取。
- 索引成功后自动入队风格提取；`file_hash` 未变化时不重复提取。
- 旧文章会后台自动分批回填；显式重建索引时自动暂停，结束后继续。
- 运行时新增 `getStyleContext()`：
  - `manual` 模式：全局画像做整体语气基线，指纹与摘录只来自手动指定文章。
  - `auto` 模式：全局画像 + 最相关 `1-2` 篇文章。
- `/api/agent/outline` 已改为直接复用 `getStyleContext()`，大纲生成与创作改写现在共用同一套风格上下文，不再单独走旧的临时风格样本分支。

### 2. 请求规划与范围识别

- 新增 `resolveCanvasRequest()`，固定输出：
  - `intent`
  - `scope_mode`
  - `target_block_ids`
  - `operation_kind`
  - `needs_style`
  - `needs_knowledge`
  - `clarify_needed`
  - `clarify_question`
- 规则优先支持：
  - `@bN`
  - `@b2 @b3`
  - `@b2-b5`
  - `第 2 段 / 第 3 块`
  - `开头 / 结尾 / 全文`
  - 上一轮目标块延续
  - 上一轮分析建议摘要延续
  - 当前文章语义匹配
- 只有规则无法稳定收束时，才触发一次低成本 `canvas_query_plan` helper。
- 助手消息 `meta` 现在固定保存 `last_focus_summary`，让“按刚才建议改”“继续改刚才那段”类请求可以沿用上一轮目标块、操作类型和建议摘要。

### 3. 创作执行链路升级

- 新增 `lib/canvasAgent.js`，用“规划器 + 执行器”替代旧的单块工具循环主链路。
- 支持 4 类输出模式：
  - 澄清追问
  - 文本回复
  - 文章分析
  - 编辑预览
- `delete` 和双段顺序互换优先走规则化处理，减少不必要的 LLM 调用。
- 全文改写固定只处理：
  - `paragraph`
  - `list`
  - `blockquote`
- 全文预算护栏：
  - 软上限 `12` 个可写块
  - 硬上限 `20` 个可写块
  - 单批最多 `4` 块
  - 预计编辑调用超过 `6` 次时直接拦截并要求缩小范围

### 4. 批量预览与恢复

- 新增 `canvas_operation_sets` 表，持久化保存一整组待应用修改。
- `/api/agent/run` 现在会：
  - 返回 `token`
  - 返回 `batch_start / batch_progress / batch_done`
  - 返回 `assistant_meta`
  - 为编辑型结果创建 `operation_set`
- `/api/agent/apply` 现在支持：
  - `{ operation }`
  - `{ operations }`
  - `cancel`
- `applyOperations()` 会按顺序在文章副本上执行；任一 `OLD_MISMATCH` 或 `BLOCK_NOT_FOUND` 都整体回滚。
- 会话详情接口现在会返回 `pending_operation_sets`，用于刷新恢复。
- 当前文章 hash 变化后，未应用预览会转为 `stale`，可查看但不能直接应用。
- `/api/agent/run`、`/api/agent/apply`、会话恢复链路已补齐 Canvas 专用日志事件，固定记录 `canvas_mode / scope_mode / operation_kind / helper_used / operation_count / fallback_reason / operation_set_status`。

### 5. 配置与遗留链路收口

- `/api/settings` 已补齐后端可配置的 Canvas 字段：
  - `enable_style_extraction`
  - `enable_article_analysis`
  - `global_edit_soft_max_blocks`
  - `global_edit_hard_max_blocks`
  - `style_extraction_model`
- 本轮不新增设置页 UI，上述配置仅后端可读写。
- 旧的 `/api/agent/intent` 已移除。
- 旧的 `lib/agent.js` Canvas 工具循环实现已移除。
- 旧的 `OperationPreview` 组件已移除。

### 5. 前端画布升级

- 创作页右侧 AI 协作区已从“单条待应用修改”升级为“多组待应用预览”。
- 支持恢复全部未应用预览。
- 支持整组应用、整组取消、过期失效和 `stale` 状态提示。
- InputBar 继续支持 `@bN`，并新增“全文”快捷项。
