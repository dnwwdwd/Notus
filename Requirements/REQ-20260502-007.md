# REQ-20260502-007 创作页新增主动提问卡片与回答后自动续跑

## 分类

- 类型：功能优化
- 状态：已完成

## 背景

创作页此前只能在意图不清楚时返回一句普通追问文本，没有独立状态，也不会冻结“上一轮已经生成的内容”。这会带来三个直接问题：

1. 用户说“把上面的内容写到文档中”时，系统容易把它继续当成普通聊天，而不是编辑请求。
2. 即使下一轮用户明确说“写到文档中”，系统也可能重新生成一轮内容，而不是把上一轮内容真正落到文档预览里。
3. 页面刷新、切换会话或文章变化后，原本的澄清信息无法稳定恢复，也无法判断卡片是否已经失效。

## 本次目标

1. 在创作页上线结构化主动提问卡片，只在编辑意图不明确但仍可继续创作时触发。
2. 冻结来源内容快照和文章版本，避免“上一条回复”在后续轮次里漂移。
3. 用户回答卡片后自动续跑生成 `operation_set` 预览，不要求再补发一轮消息。
4. 补齐卡片状态、恢复、过期、失败重试和输入框自由回答。

## 实施结果

### 1. 创作规划器升级为结构化槽位识别

- 创作页请求规划器现在会优先识别：
  - `source_content_ref`
  - `target_location`
  - `write_mode`
- 对“把上面的内容写到文档中”“把刚才生成的内容写进去”这类话术，优先按编辑请求处理，不再先退回普通聊天。
- 当缺少目标位置、写入方式、内容指代不稳定，或候选块冲突明显时，才会触发结构化提问卡片。
- 连续结构化追问超过两轮后，会退回自然语言追问，不继续堆卡片。

### 2. 新增 interaction 持久化表与恢复链路

- 新增 `conversation_interactions` 表，持久化保存：
  - 提问卡片状态
  - 来源内容快照
  - 来源内容 digest
  - 当前文章 hash
  - 结构化问题定义
  - 用户结构化回答
- 同一会话同一时刻只允许 1 张 `pending` 卡片；新卡片创建时，旧卡片会自动转为 `stale`。
- 卡片默认保存 7 天，超时后自动转为 `cancelled`。
- 会话详情接口现在会一并返回 `pending_interactions`，刷新后能恢复 `pending / stale / failed` 卡片。

### 3. 回答卡片后自动续跑，不再要求用户补发一轮

- 新增 `POST /api/interactions/[id]/respond`。
- 回答卡片成功后：
  - 后端会把结构化答案写入 `response_json`
  - 追加一条 `user` 摘要消息到对话历史
  - 返回 `should_continue` 和 `resume_payload`
- 前端收到 `should_continue=true` 后，会直接再次调用 `/api/agent/run` 续跑生成预览。
- 如果续跑失败，系统会追加一条助手提示消息，并提供“重试生成预览”按钮，不要求重新回答卡片。

### 4. “写到文档中”路径改为优先使用冻结内容

- 创建卡片时会冻结来源内容，不再依赖“最新上一条消息”。
- 回答完成后，如果本轮是把已有内容写入文档，执行器会优先使用冻结内容直接构造插入预览，不再重新生成同一段正文。
- 这条路径会再次校验文章 hash 和来源 digest，文章变化或来源失效时不会直接继续执行。

### 5. 创作页前端补齐卡片、提示与失败重试

- 当前正式口径已由 [REQ-20260518-001](/Users/hejiajun/Documents/lzc_projects/Notus/Requirements/REQ-20260518-001.md) 收口：提问卡片现在改为右侧 AI 面板底部抽屉，覆盖输入框，不再以内联助手消息形式展示。
- 抽屉保留结构化问题、自定义输入、自动续跑和失败重试；失败时只保留重试入口，不再额外插入内联重试块。
- 输入框自由回答链路仍保留在后端，但当前前台主要通过抽屉表单提交。
- 新增：
  - 结构化提问卡片
  - 回答后自动续跑
  - 续跑失败重试卡片
  - `pending / stale / failed` 卡片恢复显示

## 当前边界

1. 本轮只在创作页上线结构化提问卡片，知识库页仍保留现有文字澄清方式。
2. 回答卡片后仍只生成待应用预览，不会自动把修改写回文档。
3. 当前自由回答解析优先覆盖 `source_content_ref / target_location / write_mode` 三类槽位，其他更复杂的口语表达仍建议继续补充明确说明。
