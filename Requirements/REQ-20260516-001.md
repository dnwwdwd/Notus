# REQ-20260516-001 Workspace Agent 第一阶段改造

## 基本信息

- 日期：2026-05-16
- 分类：功能优化
- 状态：已完成
- 来源：用户提出

## 需求标题

按 `docs/Agent_upgrade_plan.md` 的方向，将 Notus 第一阶段改造成以本地 Markdown 工作区为中心的 AI 协作者。

## 背景

Notus 的长期形态已经明确为“Obsidian 式本地 Markdown 工作区 + Claude Code / Codex 式 AI 协作”。旧的“知识库问答 + 普通文件改写”不足以表达真实产品方向。本次需求要求在不清空向量索引、不批量改写旧 Markdown、不引入新数据库服务的前提下，先完成数据结构、索引同步、文档级上下文、会话范围和工具层的第一阶段改造。

## 本次目标

1. 保留 `files.id -> chunks.file_id -> chunks_vec.chunk_id` 关系，避免迁移时丢失旧向量数据。
2. `.md` 文件作为真实正文来源，数据库只保存索引、元数据、缓存和运行状态。
3. `files` 增加稳定身份、文件状态、frontmatter、标签和大纲等元数据字段。
4. `chunks` 增加 `source_hash` 与 `index_version`，用于判断 chunk 是否来自当前文件版本。
5. 会话范围拆为 `read_scope / retrieval_scope / write_scope / style_scope`。
6. 知识库问答从 chunk 命中聚合到文档，再从文件系统读取真实 Markdown。
7. 第一阶段新增工作区工具执行层，先覆盖搜索、读文件、风格上下文、提问和文章预览。
8. 写入继续采用预览确认，不做多文件后台直接写入。

## 已完成实现

1. `lib/db.js` 做非破坏性迁移，新增 `files`、`chunks`、`conversations` 元数据字段，并回填 `chunks.source_hash = files.hash`。
2. `lib/files.js` 新增 Markdown frontmatter、stable id、文件大小、mtime、token 数、tags 和 heading outline 的提取与回填；Notus 新建文件会写入 frontmatter id，旧文件不会批量写回。
3. `lib/indexer.js` 写入 `source_hash` / `index_version`，hash 和版本未变化时跳过重切分与重新 embedding。
4. `lib/watcher.js` 增加文件级串行队列、延迟删除和基于 frontmatter id 的重命名识别。
5. `lib/workspaceDocuments.js` 新增文档级检索上下文层，从 chunk/section 聚合到 `file_id` 后读取文件系统 Markdown，并标记 `stale_index`。
6. `lib/workspaceScope.js` 新增四类 scope 的规范化、校验、文件解析和读写范围组合逻辑。
7. `pages/api/conversations/[id]/scope.js` 新增会话 scope 更新与重置接口。
8. `pages/api/chat.js` 保留原 SSE 事件，同时增加文档摘要和文档统计；完整正文只进入模型上下文，不直接发送给前端。
9. `components/ChatArea/ChatMessage.js` 与 `pages/knowledge.js` 展示本次读取的 Markdown 文档摘要和完整/节选统计。
10. `lib/workspaceAgentTools.js` 新增统一工作区工具执行骨架，工具失败返回结构化 `hint`。
11. `pages/api/agent/apply.js` 在应用预览前校验 operation set 状态、`article_hash`、可选 `file_hash` 与 `write_scope`。

## 边界说明

1. 本阶段不清空 `chunks_vec`，不主动重建所有向量。
2. 已存在的 chunk 和向量仍沿用原来的整数主键关系。
3. 只有 hash 或 `index_version` 变化的文件会重新切分并重新 embedding。
4. 旧 Markdown 不会被批量写入 frontmatter id；只有 Notus 新建文件会自动写入。
5. 多文件写入仍是后续阶段能力；本阶段只保留预览结构和权限校验。
